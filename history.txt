### 🧐 Análisis del sistema Kauka (Lo que funciona bien)

1.  **Manejo de Usuarios (`kauka` vs `node` vs `root`)**:
    *   El Dockerfile crea un usuario específico (`kauka` con UID 2000). Esto es excelente para evitar conflictos de permisos en Linux cuando se montan volúmenes.
    *   Los comandos `postStartCommand` que ajustan el `chown` de `.ssh_local` son clave para que el agente SSH funcione dentro del contenedor sin romper los permisos fuera.
2.  **Entorno de Terminal (Zsh + Powerlevel10k)**:
    *   Está configurado para ser cómodo (`zsh-in-docker`, historial, autocompletado).
3.  **Inyección de Credenciales**:
    *   Monta `~/.ssh` y `~/.gitconfig` del host, lo cual es vital para trabajar con repositorios privados sin re-autenticarse.
4.  **Stack Tecnológico**:
    *   Base `node:23` (muy actual).
    *   Instalación manual de Hugo y Go (necesario para Kauka, quizás opcional para Quartz v4, pero útil tenerlo).

#### 1. Dockerfile

```dockerfile
# Use an official node image as a parent image
FROM node:23

# Argumentos para versiones (puedes actualizarlos si Quartz requiere algo específico)
ARG HUGO_VERSION=0.148.1
ARG GO_VERSION=1.24.5

# --- CAMBIO CLAVE: Usuario SOLE ---
ARG USERNAME=sole
ARG USER_UID=2000
ARG USER_GID=$USER_UID

# Install dependencies
RUN groupadd --gid $USER_GID $USERNAME && \
    useradd -s /bin/bash --uid $USER_UID --gid $USER_GID -m $USERNAME && \
    apt-get update && \
    apt-get install -y ca-certificates openssl git curl wget build-essential zsh && \
    rm -rf /var/lib/apt/lists/*

# Determine architecture
RUN ARCH=$(uname -m) && \
    if [ "$ARCH" = "aarch64" ]; then ARCH=arm64; else ARCH=amd64; fi && \
    echo "Architecture: $ARCH"

# --- NOTA: Si Quartz v4 no usa Hugo, puedes comentar/borrar este bloque ---
# Download and install Hugo Extended
RUN ARCH=$(uname -m) && \
    if [ "$ARCH" = "aarch64" ]; then ARCH=arm64; else ARCH=amd64; fi && \
    wget -O hugo_extended_${HUGO_VERSION}.tar.gz https://github.com/gohugoio/hugo/releases/download/v${HUGO_VERSION}/hugo_extended_${HUGO_VERSION}_linux-${ARCH}.tar.gz && \
    tar xf hugo_extended_${HUGO_VERSION}.tar.gz && \
    mv hugo /usr/bin/hugo && \
    rm hugo_extended_${HUGO_VERSION}.tar.gz && \
    echo "Hugo installed"

# --- NOTA: Si Quartz v4 no usa Go, puedes comentar/borrar este bloque ---
# Download and install Go
RUN ARCH=$(uname -m) && \
    if [ "$ARCH" = "aarch64" ]; then ARCH=arm64; else ARCH=amd64; fi && \
    wget -O go${GO_VERSION}.linux-${ARCH}.tar.gz https://dl.google.com/go/go${GO_VERSION}.linux-${ARCH}.tar.gz && \
    tar -C /usr/local -xzf go${GO_VERSION}.linux-${ARCH}.tar.gz && \
    rm go${GO_VERSION}.linux-${ARCH}.tar.gz && \
    echo "Go installed"

# Export Go path
ENV PATH=$PATH:/usr/local/go/bin

# Verify versions
RUN git --version && \
    hugo version && \
    go version

# Default shell
USER $USERNAME

# Set the default shell to bash rather than sh
ENV SHELL /bin/zsh

# Uses "git", "ssh-agent" and "history-substring-search" bundled plugins
RUN sh -c "$(wget -O- https://github.com/deluan/zsh-in-docker/releases/download/v1.2.1/zsh-in-docker.sh)" -- \
    -p git -p ssh-agent -p 'history-substring-search' \
    -a 'bindkey "\$terminfo[kcuu1]" history-substring-search-up' \
    -a 'bindkey "\$terminfo[kcud1]" history-substring-search-down'

ENTRYPOINT [ "/bin/zsh" ]
CMD ["-l"]
```

### ¿Por qué es mejor esto?

1.  **Permisos Linux**: Al usar el usuario `sole` (UID 1000) dentro del contenedor, los archivos que crees o edites tendrán tu mismo usuario en el sistema anfitrión. ¡Adiós a los archivos bloqueados por root!
2.  **Git Real**: Al montar tu `.gitconfig` y `.ssh`, tus commits quedarán firmados con tu usuario real automáticamente, sin tener que configurar `git config` cada vez que inicias el contenedor.
3.  **Simplicidad**: Ya no necesitas levantar 2 contenedores y una red compleja solo para editar texto. Tienes un entorno unificado.
4.  **Velocidad**: Al no usar volúmenes Docker para node_modules (que a veces son lentos en sincronización), sino trabajar directamente sobre el sistema de archivos montado (o dejar node_modules dentro del contenedor si prefieres aislamiento total), el rendimiento suele mejorar.

GitHub Copilot: El error está en el comando `postCreateCommand` dentro de tu archivo devcontainer.json.

El log muestra que falló aquí: `... && cd quartz && npm install`.
Esto sucede por dos razones probables:
1.  **`cd quartz`**: En Quartz v4, el archivo package.json está en la **raíz** del proyecto, no dentro de la carpeta `quartz`. Al intentar entrar a esa carpeta e instalar, no encuentra nada.
2.  **Copia de SSH**: Si tu carpeta de llaves SSH está vacía o no coincide, el comando `cp` falla y detiene todo el proceso.

### Solución

Abre tu archivo devcontainer.json y busca la línea `"postCreateCommand"`. Reemplázala por esta versión corregida que es más robusta y elimina el cambio de directorio innecesario:

```json
"postCreateCommand": "mkdir -p ~/.ssh && cp -r ~/.ssh-host/. ~/.ssh/ 2>/dev/null || true && chmod 700 ~/.ssh && chmod 600 ~/.ssh/* 2>/dev/null || true && npm install",
```

### ¿Qué hace esta corrección?
*   **Elimina `cd quartz`**: Ejecuta `npm install` directamente en la raíz (donde debe estar tu package.json).
*   **`|| true`**: Si la copia de llaves SSH falla (porque no hay llaves o la carpeta está vacía), ignora el error y continúa con la instalación de `npm`, evitando que se rompa todo el proceso.

Una vez guardado el cambio, vuelve a ejecutar **"Dev Containers: Rebuild Container"**.

# Usamos Node 22 como base (versión LTS reciente)
FROM node:22

# Argumentos para crear el usuario SOLE
ARG USERNAME=sole
ARG USER_UID=1000
ARG USER_GID=$USER_UID

# 1. Configuración de usuario y herramientas básicas
# FIX: Eliminamos el usuario 'node' (UID 1000) que viene por defecto para evitar conflictos
RUN if id -u node > /dev/null 2>&1; then userdel -f node; fi && \
    if getent group node > /dev/null 2>&1; then groupdel node; fi && \
    groupadd --gid $USER_GID $USERNAME && \
    useradd -s /bin/bash --uid $USER_UID --gid $USER_GID -m $USERNAME && \
    apt-get update && \
    apt-get install -y \
    git \
    curl \
    wget \
    zsh \
    python3 \
    python3-pip \
    python3-venv \
    && rm -rf /var/lib/apt/lists/*

# 2. Configuración de Zsh (Powerlevel10k y plugins)
USER $USERNAME
ENV SHELL /bin/zsh

RUN sh -c "$(wget -O- https://github.com/deluan/zsh-in-docker/releases/download/v1.2.1/zsh-in-docker.sh)" -- \
    -p git \
    -p ssh-agent \
    -p 'history-substring-search' \
    -a 'bindkey "\$terminfo[kcuu1]" history-substring-search-up' \
    -a 'bindkey "\$terminfo[kcud1]" history-substring-search-down'

# 3. Directorio de trabajo
WORKDIR /workspace
CMD ["sleep", "infinity"]

### 1. Resumen Técnico de la Migración (Para tu documentación interna)

**Objetivo:** Estandarizar el entorno de desarrollo de SOLE Voltaje basándose en la arquitectura de Kauka, eliminando la deuda técnica de Docker y simplificando la estructura de archivos de Quartz v4.

**Cambios Realizados:**

*   **Infraestructura (DevContainer):**
    *   Se eliminó `docker-compose` para desarrollo en favor de una imagen unificada (`Dockerfile` único).
    *   **Gestión de Usuarios:** Se migró de `root` al usuario `sole` (UID 1000) para coincidir con el usuario estándar de Linux/WSL y eliminar problemas de permisos en archivos generados (`root owned files`).
    *   **Inyección de Credenciales:** Se configuró el montaje automático de `~/.ssh` y `~/.gitconfig` del host al contenedor, permitiendo firmar commits y hacer push/pull sin re-autenticarse.
    *   **Stack:** Imagen base `node:22` (Bookworm) con Python 3 instalado para soporte de scripts legacy.
    *   **Terminal:** Implementación de Zsh + Powerlevel10k para mejorar la experiencia de desarrollador (DX).

*   **Arquitectura de Archivos (Refactorización):**
    *   **Flattening (Aplanamiento):** Se movió Quartz de una subcarpeta anidada (`./quartz/quartz`) a la raíz del proyecto. Ahora package.json, `quartz.config.ts` y la carpeta content viven en el primer nivel.
    *   **Separación de Responsabilidades:**
        *   content: Solo contenido Markdown (El Jardín Digital).
        *   `quartz/`: Solo código fuente del generador.
        *   `docs/`: Documentación del proyecto (Técnica y de Archivo).
        *   scripts: Scripts de utilidad (Python/Bash) centralizados.

*   **Workflow:**
    *   Se simplificaron los comandos de inicio. Ya no se requiere `cd quartz`.
    *   Comando estándar: `npm run dev` (alias de `npx quartz build --serve`).
